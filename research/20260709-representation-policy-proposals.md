# 表現ポリシーの複数案検討と批評ループの記録

- Issue: #72
- 作成日: 2026-07-09
- 入力: 表現手法ディープリサーチ(25クレーム検証済み)+ DSL/OSS 調査(`20260709-process-dsl-options.md`)
- 方法: 3案を生成し、3つの批評レンズ(新人エンジニア理解度 / docs-as-code 保守性 / AI-DLC・提案ツール拡張性)で相互批評 → 合成

## リサーチから得た設計制約(検証済み事実)

1. **Diátaxis**: 学習用(チュートリアル・解説)と実務用(ハウツー・リファレンス)は分けて書く。混ぜると両方の読者を失う
2. **DITA トピック粒度**: 「単一の問いに答えるほど短く、単独で意味が通るほど長い」— プロセス/アクティビティ/タスク単位のページ分割基準になる
3. **プレインランゲージは専門家にも有効**: GOV.UK の調査では専門家ほど平易な文章を好む(法律文書で80%)。「新人向けに崩す」のではなく「全読者のために平易にする」
4. **GOV.UK の定量規則**: 25語(日本語なら一文50〜60字目安)を超えたら分割、段落は5文まで
5. **やさしい日本語の3ステップ**: ①情報を取捨選択して整理 → ②表現を平易化 → ③対象読者で伝わるか確認。逐語変換ではなく**読者基準の情報削減**が第一原則
6. **C4 model**: 地図のズームに例えた段階的詳細化(4レベル)がドリルダウン設計の参照モデル
7. **Mermaid click**: `securityLevel: 'loose'` が必要(astro-mermaid の mermaidConfig で設定可能)
8. **単一既製 DSL では不成立**(DSL調査): BPMN は知識ベース語彙がなく、SPEM はツール死亡、実行系 DSL は人間系語彙がない → データ層は自作スキーマが合理的

## 3案

### 案A: 読み物主導型(Diátaxis 解説ページ+図は挿絵)

各プロセスを平易な解説記事として執筆し、図は本文の補助に留める。ページ構成は Diátaxis の explanation に寄せる。

### 案B: データ駆動ドリルダウン型(スキーマ+3ズームレベル)

プロセスを YAML スキーマ(SPEM 簡約語彙)で定義し、C4 の地図アナロジーに倣った3ズームレベル(L1 全体マップ → L2 フェーズ → L3 個別作業)をデータから生成する。図のノードは click リンクで下位レベルのページへ遷移する。

### 案C: ビューア埋め込み型(bpmn-js アイランド)

BPMN 2.0 でプロセスを記述し、bpmn-js の NavigatedViewer をアイランド埋め込みして、ビューア内サブプロセス展開でドリルダウンする。

## 批評ループ(3レンズ × 3案)

| レンズ | 案A 読み物主導 | 案B データ駆動 | 案C bpmn-js |
| --- | --- | --- | --- |
| 新人理解度 | ○ 文章は読みやすいが、**全体像の把握が読者の脳内構築に依存**し「今どこ」を見失う(認知負荷大) | ◎ 地図型ズームで「今どこ」が常に明示される。ただし**データから生成した図だけでは"なぜ"が伝わらない**(解説文の設計が別途必要) | △ 図は本格的だが BPMN 記法自体が新人には未知の語彙(学習コストの転嫁) |
| docs-as-code 保守性 | △ 8プロセスを手書きすると表現が発散し、比較可能性が崩れる(#20 の目的と矛盾) | ◎ スキーマ検証を `npm run check` に統合でき、textlint と同じ「強制層」に載る。**ただしスキーマ設計ミスが全ページに波及**するリスク | × XML は差分レビュー不能、編集に GUI ツール必須、ウォーターマーク常時表示 |
| ツール拡張性 | × 文章からの知識抽出は再構造化が必要(手戻り最大) | ◎ 同じデータが提案ツール(M7)の知識ベースにそのまま転用できる**唯一の案** | △ XML から別途抽出層が必要 |

### 批評から得た修正点(ループ2周目)

- 案B単独の弱点「図だけでは"なぜ"が伝わらない」は、**案Aの記述原則を各レベルのページ本文に組み込む**ことで解消する: L1〜L3 の各ページは「図(構造)+平易な解説(なぜ・何のため)+原典対照(出典)」の3点セットとする(二重符号化: 図と言葉の併用)
- 案Bのスキーマ波及リスクは、`schemaVersion` の導入と「最初は2プロセスで検証してから展開」で緩和する(DSL調査の推奨)
- 案Cは棄却するが、**特定プロセスの詳細フローで表現力不足になった場合の将来オプション**として残す(データと描画を分離しておけば差し替え可能)

## 合成結果(採用案)

**案B(データ駆動3ズームレベル)を骨格に、案Aの記述原則を各ページに埋め込むハイブリッド。**
決定の詳細は ADR-0006(表現ポリシー)と ADR-0007(スキーマ駆動)に記録する。

- 構造: L1 全体マップ / L2 フェーズ / L3 個別作業 の3ズームレベル。図のノードクリックで下位ページへ遷移
- 1図あたりノード15個前後を上限とし、超えたら分割(認知負荷とMermaidレイアウト限界の両方に対応)
- 文章: 本文は一文60字目安・段落5文まで。原典の難解な定義は「平易な言い換え → 原典引用(出典付き)」の対で載せる(やさしい日本語3ステップの①②を適用し、③はコミュニティフィードバックが担う)
- データ: 自作 YAML スキーマ(SPEM 簡約語彙 = Process/Phase/Activity/Task + Role/WorkProduct/Gate)+ Astro Content Collections。調査フレームワークの6+2要素と同型
