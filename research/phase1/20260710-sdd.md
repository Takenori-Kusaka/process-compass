## 0. プロセスの概要

### 0-0. 最重要の前提: 「SDD」という語は新旧2つの文脈で使われる

「仕様駆動開発(SDD)」は近年2つの異なる意味で使われる。本メモは両者を明確に区別する。本プロジェクトの主題(生成AI時代の開発プロセス)に直結するのは **B) の新しい SDD** であり、本メモの記述の中心もそこに置く。

- **A) 古典的な意味(仕様先行の系譜)**: 仕様(スペック)を先に厳密に定義してから実装する、という古くからある考え方。形式仕様記述(Z, VDM, TLA+ 等)、契約による設計(Design by Contract, Bertrand Meyer)、仕様ベーステストなどの系譜。ここでの「仕様」は主に**形式的・数学的な厳密さ**を志向する。
- **B) 2025年前後の新しい意味(生成AIコーディング時代の Spec-Driven Development)**: 自然言語(主に Markdown)で書いた仕様書を、AI コーディングエージェントへの**唯一の信頼できる情報源(single source of truth)**として与え、AI がそこからコードを生成する開発手法。GitHub の Spec Kit、AWS の Kiro が代表実装。「vibe coding(場当たり的なプロンプト生成)」への対抗概念として2025年に急速に注目された。
  - 出典: [github/spec-kit README](https://github.com/github/spec-kit)、[spec-kit: spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)、[Kiro Docs: Specs](https://kiro.dev/docs/specs/)

### 0-1. 新しい SDD(B)の基本情報

- 提唱者・原典・成立年: 単一の提唱者はいない。2025年に複数の潮流が同時発生した「運動」に近い。主要な原典・起点は以下。
  - **GitHub Spec Kit**(オープンソースのツールキット、2025年公開)。SDD を「開発の中心を仕様に置く方法論」と定義し、`Specify → Plan → Tasks → Implement` の4フェーズを実装。
    - 出典: [github/spec-kit](https://github.com/github/spec-kit)、[spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)、[Spec Kit Documentation](https://github.github.com/spec-kit/)
  - **AWS Kiro**(agentic IDE、2025年)。`requirements.md → design.md → tasks.md` の3フェーズ「Feature Spec」を実装。EARS 記法による構造化要件が特徴。
    - 出典: [Kiro Docs: Specs](https://kiro.dev/docs/specs/)、[Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/)
  - **Sean Grove(OpenAI)の講演 "The New Code"**(AI Engineer World's Fair 2025)。「仕様(specifications)こそがプログラミングの基本単位であり、コードは仕様からの lossy な射影にすぎない」という思想的支柱を提示。
    - 出典: [The New Code — Sean Grove 講演トランスクリプト](https://lawwu.github.io/transcripts/8rABwKRsec4.html)
- 一言でいうと: **自然言語の仕様書を「真実の源」として先に固め、計画→タスク分解→実装の各段階を構造化された Markdown 成果物として積み上げ、コードの生成を AI エージェントに委ねる開発手法**。人間の役割は「仕様を書き・レビューし・承認すること」に移る。
  - 出典: [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)
- ISO/IEC/IEEE 12207:2017 の4プロセス群のうち厚くカバーする領域(解釈): テクニカルプロセス群の上流(**要件定義・アーキテクチャ設計・設計**)を Markdown 成果物として明示化し、それを**実装(Implementation)への機械翻訳の入力**にする点が本質。合意/組織/マネジメント系プロセス群は原典では薄い。

### 0-2. 位置づけの注意(粒度)

- 新しい SDD は「ライフサイクル(組織的プロセス)」ではなく、**個々の機能(feature)を作る単位のミクロ〜メゾなワークフロー**として定義されている。Kiro も Spec Kit も「1機能=1スペック(1ディレクトリ)」を基本単位とする。スクラム等の上位ライフサイクルの「内側」で、実装の進め方として組み込む位置づけ。
  - 出典: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/)
- ただし古典的 SDD(A)は「上流工程の思想」であり、ウォーターフォールの要件・設計フェーズと親和する。粒度が異なる点に注意。

## 2. 階層構造(Process → Phase → Activity → Task)

<!-- 後で図解しやすいよう入れ子で。ゲートは「仕様承認」を人間が行う decision 型として表現できるのが新SDDの核心 -->

- **全体プロセス(Process): 仕様駆動開発**
  - **Phase 1: 仕様定義(Specify / Requirements)**
    - Activity: 機能の目的・前提・制約・成功基準を自然言語で記述させる
      - Task: 人間が「何を作りたいか」を AI に伝える(`/speckit.specify` 実行)
      - Task: AI が構造化された仕様書(spec.md / requirements.md)を自動生成。ユーザーストーリーと受け入れ基準を含む。Kiro は **EARS 記法**(`WHEN [条件] THE SYSTEM SHALL [期待動作]`)で記述
      - Task: 曖昧点を AI が明示(Spec Kit は `[NEEDS CLARIFICATION]` マーカー)
      - **Gate: 仕様承認(人間が decision)** ← 新SDDの核心。人間が要件をレビューし、次フェーズへ進めるか判断
  - **Phase 2: 計画・設計(Plan / Design)**
    - Activity: 承認された仕様を技術的実装計画・アーキテクチャに変換
      - Task: AI がデータモデル・API 契約・技術選択・シーケンス図を生成(`/speckit.plan`、design.md)
      - Task: Spec Kit は「憲法(constitution.md)」との整合性ゲート(Simplicity/Anti-Abstraction/Integration-First)をチェックリストで確認
      - **Gate: 設計承認(人間が decision)**。技術選択と設計の妥当性を人間がレビュー・承認
  - **Phase 3: タスク分解(Tasks)**
    - Activity: 計画を実行可能な離散タスクへ分解
      - Task: AI がタスクリスト(tasks.md)を生成。依存順に整列し、並列可能タスクを `[P]` でマーク、各タスクにファイルパス・入出力・依存を付与
      - **Gate: タスク承認(人間が decision)**
  - **Phase 4: 実装(Implement)**
    - Activity: AI エージェントがタスクに沿ってコードを生成
      - Task: 依存を解析し独立タスクを並列(Kiro は "wave" 単位で並列実行)
      - Task: Spec Kit の憲法(Article III)では「テストを先に書き、人間が承認し、失敗を確認してから実装コードを書く」順序を強制(TDD 内包)
      - **Gate: テスト承認(人間が decision、Spec Kit)**。実装前に人間がテストを検証・承認
  - **Phase 5: 検証(Verification / 反復)**
    - Activity: 生成物が仕様を満たすか検証し、乖離があれば仕様へ戻る
      - Task: 受け入れ基準に対する検証、仕様とコードの相互整合チェック(cross-artifact analysis)
    - 注: Spec Kit/Kiro とも「仕様が真実」なので、修正はコードでなく**仕様に立ち返って再生成**するのが原則

## 3. roles(ロール)

新しい SDD の核心は **「仕様を書く主体=人間」「コードを書く主体=AI」への役割の分割**。これは本プロジェクトの「AI が外注(実装委託)相当を担う」議論に直結する。

| ロール | 責任 | 兼務の実態(日本の現場) | 出典 |
| --- | --- | --- | --- |
| 仕様の著者(人間) | 意図・要件・制約・成功基準を記述する。曖昧性を解消し仕様を確定させる。SDD では最も価値の高い活動 | 多くは1人の開発者/PM が兼務。専任の仕様担当は稀 | [The New Code](https://lawwu.github.io/transcripts/8rABwKRsec4.html)、[spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md) |
| 承認者(人間) | 各フェーズ成果物(仕様/設計/タスク/テスト)をレビューし次段階へ進める decision を下す | 仕様著者と同一人物が兼ねることが多い(セルフレビュー化のリスク) | [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/) |
| AI コーディングエージェント | 仕様→計画→タスク→コードの各変換を実行。実装の主たる担い手 | Copilot / Claude / Gemini / Codex / Kiro など。Spec Kit は30以上のエージェントに対応 | [github/spec-kit](https://github.com/github/spec-kit) |
| ステークホルダー(非技術者含む) | 仕様が自然言語 Markdown なので、非エンジニアもレビュー・貢献できる | 制度としては未成熟。実態は開発者主導 | [The New Code](https://lawwu.github.io/transcripts/8rABwKRsec4.html) |

- Sean Grove の主張: 開発者が生む価値の **10〜20% がコード、80〜90% は構造化されたコミュニケーション(要件把握・計画・検証)**。AI が進むほど「意図を明確に articulate できる人=最も価値ある programmer」になる。
  - 出典: [The New Code](https://lawwu.github.io/transcripts/8rABwKRsec4.html)

  - 出典: [Spec Kit Documentation](https://github.github.com/spec-kit/)

## 5. gates(ゲート・決裁)

新SDDの特徴は、各フェーズ間に **人間が下す decision 型ゲート(仕様承認・設計承認・タスク承認)** が明示的に配置される点。これが vibe coding(ゲートなしの即時生成)との決定的な違い。

| ゲート名 | Deliverables | Criteria(判定基準) | 判定者 | 判定結果の種類 | 出典 |
| --- | --- | --- | --- | --- | --- |
| 仕様承認 | spec.md / requirements.md | 意図と要件が正確・完全か、曖昧点が解消されたか | 人間(仕様著者/レビュア) | 承認 / 修正差し戻し | [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/) |
| 設計承認 | plan.md / design.md | 技術選択の妥当性、憲法/steering との整合 | 人間 | 承認 / 差し戻し | [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md) |
| タスク承認 | tasks.md | 分解粒度・依存・網羅性が妥当か | 人間 | 承認 / 差し戻し | [Kiro Docs](https://kiro.dev/docs/specs/) |
| テスト承認(Spec Kit) | テストコード | テストが要件を表現し、かつ失敗することを確認 | 人間 | 承認 / 差し戻し(実装はこの後) | [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)(Article III) |
| Constitutional Gate(Spec Kit, Phase -1) | plan.md 内チェックリスト | Simplicity / Anti-Abstraction / Integration-First | 人間(チェックリスト確認) | 準拠 / 是正 | [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md) |
| (ゲートを外すモード)Kiro "Quick Plan" | 3成果物を一括生成 | ― | なし(without approval gates) | 自動進行 | [Kiro Docs: Specs](https://kiro.dev/docs/specs/) |

- 重要な含意: Kiro には「よく理解された機能では承認ゲートを飛ばす **Quick Plan**」が用意されている。つまりゲートは**運用者が着脱できる**。着けば「仕様先行の統制」、外せば vibe coding 寄りになる。ゲートの有無が本手法の統制強度を決める設計変数。
  - 出典: [Kiro Docs: Specs](https://kiro.dev/docs/specs/)

## 6. レビュープロセス

- **誰が**: 主に人間(多くはセルフレビュー、またはチームレビュー)。仕様が自然言語 Markdown なので、非技術ステークホルダーもレビュー可能(建前)。
- **何を**: 各フェーズ成果物(仕様→設計→タスク→テスト)。前段の成果物が次段の入力になるため、レビューは各ゲートで段階的に行われる。
- **どの形式で**: Markdown 文書のレビュー(diff レビュー可能)。Spec Kit は「cross-artifact analysis(成果物横断の整合性チェック)」やチェックリストを標準提供。
- **ゲートとの関係**: レビュー=各ゲートの判定行為そのもの。レビューとゲートが一体。
- Sean Grove の視点では、Model Spec のように**各条項に一意 ID を振り、対応するテストプロンプト(=ユニットテスト)で仕様への適合を自動検証**する構想もある(仕様の「実行可能化」)。
  - 出典: [The New Code](https://lawwu.github.io/transcripts/8rABwKRsec4.html)、[spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)

## 7. 日本的観点(標準がカバーしない領域)

- **稟議・多段決裁の有無と形**: 新SDDの原典に稟議は存在しない。ただし SDD の「フェーズごとの人間承認ゲート」は、日本企業の**上流工程での多段レビュー・承認文化と構造的に親和**しうる(要件定義書レビュー→設計レビューの流れに写像可能)。フェーズ3(将来設計)で「稟議をゲートに写像する」論点になりうる。
- **品質保証部門など第三者レビューの制度**: 原典では第三者QAは規定されない。仕様著者=承認者のセルフレビューになりがちで、**第三者レビューの空白**が生じやすい。
- **受発注構造とレビューの二重性**: 新SDDは「AI が外注(実装委託)相当を担う」構図とみなせる。**人間=発注者/仕様の書き手、AI=受注者/実装者**。従来の外注における「仕様書=発注仕様=検収基準」の役割を、そのまま AI への指示書に転用できる。日本の SIer の要件定義書・外部設計書の文化と接続しやすい。
- **建前と実運用の乖離(形骸化パターン)**:
  - 建前=「仕様が真実、コードは従属」。実運用=Kiro の Quick Plan のようにゲートを飛ばして vibe coding 化する / 生成後にコードを直接いじって仕様と乖離する、という**逆流(仕様の陳腐化)**が起きやすい。
    - 出典: [Algomatic Tech Blog: 話題のSDDを試してみた](https://tech.algomatic.jp/entry/2025/09/22/143931)
  - 日本の実践レビューでは「形式的な仕様が実際のプロジェクト実行と乖離する(implementation gap)」「AI 依存で開発者の実装力が痩せる」「ツールが未成熟で安定性に難」といった課題が指摘されている。
    - 出典: [Algomatic Tech Blog](https://tech.algomatic.jp/entry/2025/09/22/143931)
- **ロールの兼務の実態**: 仕様著者・承認者・実装レビュアが1人に集約されがち。SDD の統制力はゲートに依存するが、そのゲート判定者が仕様の書き手と同一だと統制が形骸化する。
- **日本発ツール**: 日本人エンジニア開発の OSS **cc-sdd** が登場。Kiro に近い思想で日本語仕様記述に最適化、12言語対応(Spec Kit は英語のみ)。日本の現場実情を踏まえた設計を標榜。
  - 出典: [ディアシステム Tech Blog: AI時代の仕様駆動開発入門](https://techblog.dearsystem.jp/blog/2026-05-18-01/)、[EDITH by K.S.ロジャース: cc-sddとspec-kitのアーキテクチャ比較](https://note.com/ks_rogers_note/n/n0d9292ad0150)

## 8. 考察(事実と分離)

- **新旧SDDの系譜関係(解釈)**: 古典的SDD(A、Design by Contract/形式手法)と新SDD(B)は「仕様を実装より先に厳密化する」思想を共有するが、**仕様の言語が根本的に違う**。古典は形式言語(数学的・機械検証可能)、新SDDは自然言語 Markdown(人間可読・AI解釈可能)。新SDDは「AIが自然言語を解釈できるようになった」ことで、形式手法が担っていた「厳密仕様→実装」の橋渡しを**自然言語で近似的に**実現しようとする試み、と位置づけられる。Sean Grove の「仕様に一意IDを振りテストで検証」は、Design by Contract の事前条件/事後条件を自然言語+テストで再発明した側面がある(解釈)。
- **本プロジェクトへの直結点(解釈)**: 新SDDの「人間=仕様、AI=実装」の分担は、本プロジェクトの中核テーマ「AIが外注相当を担う開発プロセス」の**具体的な参照実装**になる。特に「decision 型ゲート(仕様承認)を人間が握る」構造は、AI に実装を委ねつつ人間が統制を保つモデルとして、フェーズ3の提案設計で有力な骨格になりうる。
- **フェーズ3で効きそうな論点(解釈)**:
  - ゲートの着脱(Kiro Quick Plan)= チーム成熟度・機能の重要度に応じた「統制強度の可変設計」。Process Compass の「チーム体制・事業フェーズに合わせて最適プロセスを提案」に直接使える設計軸。
  - 仕様の逆流(コード直接編集による陳腐化)をどう防ぐかが運用上の最大の穴。
  - 第三者レビューの空白 → 日本のQA部門文化を「AI生成物の第三者検証ゲート」として組み込む提案余地。
- **TDD との関係(事実+解釈)**: Spec Kit の憲法 Article III は TDD(テスト先行・人間承認・失敗確認)を内包する。SDD は上流(仕様)から下流(テスト・実装)までを一気通貫で構造化し、その最下層に TDD を埋め込む「入れ子」構造とみなせる。
