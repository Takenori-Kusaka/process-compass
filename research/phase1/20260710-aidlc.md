## 0. プロセスの概要

### 0-0. 本メモの位置づけ(重要)

- 本プロジェクト(Process Compass)にとって **AIDLC は「最終的に現実へ落とし込もうとしている理想像(To)」の中核**である。ただし本メモの目的は、その理想像が **現在世の中でどう語られているか(From 側の言説)** を整理することにある。
- したがって記述は次の2層に厳密に分ける:
  - **建前(理想像)**: 提唱者(特に AWS)が主張する「AIエージェントが要件からデプロイまでを回し、人間は意思決定に集中する」姿。§1〜§6。
  - **現実の懸念**: 組織的責任の所在・品質保証・技術負債・ハルシネーション・セキュリティ・「人が管理できる賢さの上限」・日本の受発注/決裁構造との衝突。§7・§8。
- **用語の注意**: 綴りが3系統ある。
  - **AI-DLC** = AWS が2025年に提唱した固有の方法論(本メモの主対象)。出典: [AWS DevOps Blog "AI-Driven Development Life Cycle"](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)
  - **AIDLC / AI駆動開発ライフサイクル** = 本プロジェクトの Issue 表記。一般名詞的に「AIが主導する開発ライフサイクル」を指すときに使う。
  - **AI-native / AI-augmented SDLC** = 各社・各論者が使う類義の総称(後述 §0-3)。

### 0-1. 基本情報

- 提唱者・原典・成立年: **AWS(Amazon Web Services)**。原典ブログは **2025年7月31日公開**、著者は **Raja SP(AWS Principal Solutions Architect、Developer Transformation Programs リード)**。2025年11〜12月の **re:Invent 2025 セッション DVT214「Introducing AI-Driven Development Lifecycle (AI-DLC)」** で公式に大きく打ち出された。
  - 出典: [AWS DevOps Blog "AI-Driven Development Life Cycle: Reimagining Software Engineering"](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[re:Invent 2025 DVT214 セッションメモ(dev.to)](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)
- 一言でいうと(AWS の建前): **AIエージェントが計画・要件・ストーリー・コード・テスト・インフラ構成の生成をライフサイクル全段で「オーケストレーション」し、人間は各要所で検証・承認・重要判断を担う、AIを中心に据えて SDLC を作り直した方法論**。
  - AWS 自身の表現:「AI agents orchestrate the whole process, generating plans, code, tests, and infrastructure configurations at each stage, but humans verify and approve before execution proceeds」
  - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)
- 2つの柱(AWS 定義):
  1. **AI Powered Execution with Human Oversight**(AIが実行、人間が監督)—「AI systematically creates detailed work plans, actively seeks clarification and guidance, and defers critical decisions to humans」
  2. **Dynamic Team Collaboration**(動的なチーム協働)—個人の孤立作業から、リアルタイム協働空間へ
  - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)
- ISO/IEC/IEEE 12207:2017 の4プロセス群との対応(解釈): テクニカルプロセス群(要件〜実装〜運用)を**全域でAIに再配分**しようとする点が最大の特徴。合意・組織・マネジメント系(責任・決裁・品質保証)は AWS の言説では相対的に薄く、ここが §7・§8 の懸念点になる。

### 0-2. 実装(重要): 方法論とツールの2層

- AI-DLC は「概念(方法論)」と「実装(ワークフロー・ステアリングルール)」の2層で語られる。
  - **概念層**: 上記ブログ / re:Invent の方法論。
  - **実装層**: **awslabs/aidlc-workflows**(GitHub、ライセンス **MIT-0**、AWS Labs 維持)。AIコーディングエージェント向けの **adaptive workflow steering rules**(markdown 形式のルール群)。Kiro / Amazon Q Developer / Cursor / Cline / Claude Code / GitHub Copilot / OpenAI Codex など複数エージェントに適用可能。v1.0.1 が安定版、v2.0 がプレビュー。
  - 特徴:「only executes stages that add value to your specific request」(価値を生む段階だけ実行=適応的)、「structured multiple-choice questions in files, not chat」(会話でなくファイル上の構造化選択式質問で人間の意思決定を引き出す)。
  - 出典: [GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)、[AWS DevOps Blog "Open-Sourcing Adaptive Workflows for AI-DLC"](https://aws.amazon.com/blogs/devops/open-sourcing-adaptive-workflows-for-ai-driven-development-life-cycle-ai-dlc/)

### 0-3. 類義概念との異同(各社・各論者)

- **AI-native SDLC / AI-first**: 「AIを既存プロセスに後付け(retrofit)せず、AIを前提に作り直す」思想の総称。AWS も「We don't retrofit AI into existing agile. It's a newly imagined method」(Anupam Mishra, re:Invent)と明言し、この系譜に立つ。出典: [re:Invent 2025 DVT214 メモ](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)
- **AI-augmented SDLC**: AIを「補助(assistant)」として既存 SDLC に足す立場。AI-DLC はこれを明確に否定し、AIを「central collaborator and teammate(中心的な協働者・チームメイト)」と位置づける。出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)
- **SDD(仕様駆動開発 / Spec-Driven Development)**(本プロジェクト Issue #27): コードの前に仕様(spec.md / plan.md / tasks.md 等)を明文化し、それをAIへの指示基準にする手法。代表実装は **GitHub Spec Kit** と **Kiro**。**AI-DLC と SDD は排他ではなく入れ子**: AI-DLC は「AIが主導する仕様駆動開発」をライフサイクル全体へ拡張した上位方法論とみなせ、AI-DLC の Inception フェーズ(要件→ストーリー→ユニット化)は SDD の spec 生成に相当する。相違は、Spec Kit が spec ファイル起点でスコープが実装寄りなのに対し、AI-DLC は Mob(チーム協働)・bolts(時間箱)・Operations まで含む**組織プロセス寄り**な点。
  - 出典(二次): [サーバーワークス「AI-DLCワークフロー vs Spec Kit を使い比べてみた」](https://blog.serverworks.co.jp/ai-dlc-vs-spec-kit-comparison)、[Findy Team+「仕様駆動開発(SDD)とは」](https://jp.findy-team.io/blogs/sdd/)、[クラウドエース「仕様駆動開発(SDD)とは」](https://cloud-ace.jp/column/detail546/)

## 2. 階層構造(process → phase → activity → task)

<!-- 全体プロセス → AIが回すライフサイクル段階(Phase)→ Activity → Task。後で図解しやすい入れ子 -->

- **全体プロセス: AI-DLC(AIが主導し人間が承認するループを全段で反復)**
  - **共通ループ(全フェーズに埋め込まれる基本パターン)**: 「AI creates plans → AI seeks clarification → humans make critical decisions → AI implements plans」を SDLC の各アクティビティで反復。
    - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)
  - **時間箱の単位=「Bolt(ボルト)」**: 従来の2〜4週の Sprint を、**数時間〜1日以内の短く濃密なサイクル「bolts」**に置換。「sprints should be hours or less than a day」。
    - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[re:Invent DVT214 メモ](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)

  - **Phase 1: Inception(構想)** — ビジネス意図を要件・ストーリー・ユニット(units of work)へ変換
    - Activity: 要件分析 / ユーザーストーリー化 / アプリケーション設計方針 / リスク評価
    - Activity: **Mob Elaboration**(モブ精緻化)— チーム全員でAIの問い・提案をリアルタイム検証
      - Task: AIがビジネス意図を精緻化(refine intention)
      - Task: AIがストーリーを生成、参加者(PM/開発/QA/運用)が洞察と検証を提供
      - Task: 「4時間〜半日で、作るものとストーリーに全員が合意」して終える。AIも豊富なコンテキストを獲得
    - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)、[re:Invent DVT214 メモ](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)

  - **Phase 2: Construction(構築)** — 検証済みコンテキストを使いAIが論理アーキテクチャ・ドメインモデル・コード・テストを提案
    - Activity: 詳細コンポーネント設計 / コード生成 / テスト戦略・生成 / 品質保証
    - Activity: **Mob Construction**(モブ構築)— 技術判断・アーキテクチャ選択にチームがリアルタイムで指針を与えるチェックポイント
      - Task: AIが実装案を生成、要所で「重要判断を明示的に人間に上げる(mandatory human checkpoints)」
      - Task: 承認された計画のみ実行、成果物を再度ステークホルダーが検証
    - 出典: [AWS DevOps Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)

  - **Phase 3: Operations(運用)** — 蓄積コンテキストでAIがIaC・デプロイ・監視を管理(チーム監督下)
    - Activity: デプロイ自動化 / 監視セットアップ / 本番レディネス / (将来)インシデント対応
    - **注記**: 現行スペックでは Operations は「placeholder(将来実装の枠)」であり、Inception/Construction ほど作り込まれていない。
    - 出典: [GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)、[Micah Walter "Notes on exploring AI-DLC"](https://www.micahwalter.com/posts/notes-on-exploring-ai-dlc)

- **適応的実行(Adaptive Workflow)**: 全段を一律に回すのではなく、リクエストに価値を生む段階だけ実行し、追加拡張(security / testing / resiliency)は要件分析時にオプトインで積む。
  - 出典: [GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)

## 3. roles(ロール): 人間(意思決定)× AIエージェント(実行)の分担

| ロール | 責任(建前) | 兼務/実態の懸念 | 出典 |
| --- | --- | --- | --- |
| **AIエージェント(実行主体)** | タスク分解、代替案生成、要件/ストーリー/コード/テスト/IaC の生成、リファクタリング、ワークフローのオーケストレーション | 生成物の正しさは保証されない(ハルシネーション)。責任主体になれない | [AWS Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[dev.to "Why AI-DLC Exists"](https://dev.to/aws-heroes/why-ai-dlc-exists-reimagining-software-development-in-an-ai-first-world-4f20) |
| **人間(意思決定主体、総称)** | アーキテクチャ判断、ビジネス整合、リスク評価、倫理・品質判断、重要判断の承認。「AI may drive workflows, but humans remain responsible」 | 生成速度に対し人間の検証が追いつかず「passive approver(受動的承認者)」化するリスク | [dev.to "Why AI-DLC Exists"](https://dev.to/aws-heroes/why-ai-dlc-exists-reimagining-software-development-in-an-ai-first-world-4f20)、[AWS for Industries](https://aws.amazon.com/blogs/industries/ai-driven-development-lifecycle-for-financial-services/) |
| Product Manager | Mob セッションでビジネス意図・優先度を提示、WHAT/WHY を確定 | 従来のPO/PdM兼務。日本では事業部門と情シスの分離で不在になりがち | [re:Invent DVT214 メモ](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b) |
| Developer | Mob で技術判断・実装レビュー、AIへの技術的指針 | 役割が「書き手」から「検証者・判断者」へ移行 | 同上 |
| QA | Mob で品質観点の検証、テスト妥当性の確認 | 独立第三者検証との整合が未定義(§7) | 同上 |
| Operations | Mob で運用/インフラ観点の提示 | Operations フェーズ自体が未成熟 | 同上 |

- **役割分担の一言スローガン(建前)**: 「人は WHAT と WHY を、AI が HOW を」。AWS Heroes の整理では人間=Architectural judgment / Business alignment / Risk assessment / Ethical and quality decisions、AI=Task decomposition / Generating alternatives / Writing tests and boilerplate / Optimizing and refactoring。
  - 出典: [dev.to "Why AI-DLC Exists"](https://dev.to/aws-heroes/why-ai-dlc-exists-reimagining-software-development-in-an-ai-first-world-4f20)
- **重要な特徴(建前)**: Mob セッションは「product managers, developers, QA, and operations all in the same room」。従来SDLCの逐次ハンドオフ(sequential handoffs)を排し、全ロールを同期させる点が生産性向上の主張根拠。
  - 出典: [re:Invent DVT214 メモ](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)

## 5. gates(ゲート・決裁): Human-in-the-loop 承認点

各ゲートを Deliverables / Criteria / 判定者 / 判定結果で記述。AI-DLC のゲートは伝統的なDR/決裁より軽量な「**mandatory human checkpoint(必須の人間チェックポイント)**」として、bolt 内に高頻度で埋め込まれるのが特徴。

| ゲート名 | Deliverables | Criteria(判定基準) | 判定者 | 判定結果の種類 | 出典 |
| --- | --- | --- | --- | --- | --- |
| Mob Elaboration 合意点 | 要件・ストーリー・ユニット | 「作るもの/ストーリーに全員が合意」 | Mob参加者(PM/Dev/QA/Ops) | 合意して次へ / 差し戻し(course correction) | [AWS Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[re:Invent DVT214](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b) |
| 計画承認(plan-verify) | AIが提示した実行計画 | 「AI creates plan → humans validate the plan」 | 人間(担当ロール) | 承認して実行 / 修正指示 | [re:Invent DVT214](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b) |
| Mob Construction チェックポイント | アーキ/ドメインモデル/コード/テスト案 | 技術判断・アーキ選択の妥当性 | Dev中心のMob | 承認 / 技術指針を与えて再生成 | [AWS Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/) |
| 成果物最終検証 | 生成された最終成果物 | ステークホルダーによる再検証 | ステークホルダー | 受理 / 差し戻し | [note.com AI-DLC完全ガイド(二次)](https://note.com/cc_works_cl/n/n673726233c57?hl=en) |
| (規制業界)承認+トレーサビリティ監査 | コード〜要件の連鎖 | steering files 準拠、監査可能性 | 人間(監査/責任者) | 承認 / 是正 | [AWS for Industries](https://aws.amazon.com/blogs/industries/ai-driven-development-lifecycle-for-financial-services/) |

- **設計思想(建前)**: 「mandatory human checkpoints are built into the workflow, and all critical decisions are surfaced explicitly rather than silently absorbed into AI output」= 重要判断がAI出力に暗黙に飲み込まれないよう、明示的に人間へ上げる。「The agent proposes, the human approves」。
  - 出典: [GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)
- **決裁型ではなく検査/合意型**: これらは Go/Kill の事業決裁ゲートではなく、生成物の inspection と合意形成のチェックポイント。伝統的マイルストーン決裁(予算・GA判断)との接続は AI-DLC 言説では未定義(§7・§8)。

## 6. レビュープロセス

- **形式**: 「Mob(モブ)」= 関係ロール全員が同じ場でリアルタイムにAI生成物を検証・指針提示する集団レビュー。逐次のドキュメント回覧レビューを、同期セッションに置換する。
  - 出典: [AWS Blog](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)、[re:Invent DVT214](https://dev.to/kazuya_dev/aws-reinvent-2025-introducing-ai-driven-development-lifecycle-ai-dlc-dvt214-32b)
- **自己検証**: AI自身が包括的テストで自らの成果を検証・自己修正(steering files からの逸脱含む)。「comprehensive testing, helping AI to validate its own work and self-correct when tests fail」。
  - 出典: [AWS for Industries](https://aws.amazon.com/blogs/industries/ai-driven-development-lifecycle-for-financial-services/)
- **品質統制の量的整合(建前)**: 「増えたAI生成コード量に比例して自動品質統制も増やす」。
  - 出典: [AWS for Industries](https://aws.amazon.com/blogs/industries/ai-driven-development-lifecycle-for-financial-services/)
- **レビュー=ゲートの関係**: AI-DLC ではレビュー(Mob)とゲート(チェックポイント)がほぼ一体。独立した第三者検証(品質保証部門・監査)は steering files と監査トレーサビリティで代替する構図で、**組織的な独立レビュー機構そのものは方法論に内在しない**(§7)。

## 7. 日本的観点/現実の懸念(標準・提唱者言説がカバーしない領域)

### 7-1. 理想像が暗黙に仮定している前提条件(これが崩れると成立しない)

AWS/提唱者の言説から逆算した、「AIが自律的に回る」ための暗黙の前提:

- **高品質なモデル**: 生成物が実用水準に達するLLM/エージェント能力。ハルシネーション率が人間の検証で吸収できる範囲であること。
- **明確な仕様/意図**: Inception で「作るものに全員が合意」できるだけ、ビジネス意図が言語化可能であること(レガシー・暗黙知の多い現場ほど崩れる)。
- **検証の自動化**: 「AIが自らテストで検証・自己修正」が機能する前提=包括的な自動テストとCI基盤が既にあること。テストの正しさ自体は誰が保証するかは未解決。
- **信頼できるガードレール**: steering files が組織標準・規制(EU AI Act等)・セキュリティを正しく符号化できていること。
- **人間の検証キャパシティ**: 生成速度(bolts=時間単位)に人間の判断が追随できること。ここが「人が管理できる賢さ/速さの上限」論点。
- 明示的前提として dev.to Heroes 記事は成功要件に「Clear ownership(明確な責任者)」「Strong quality gates(強い品質ゲート)」「Responsible AI usage」を挙げ、レガシー制約・不完全なドキュメント・本番リスクの中で運用する現実を認める。
  - 出典: [dev.to "Why AI-DLC Exists"](https://dev.to/aws-heroes/why-ai-dlc-exists-reimagining-software-development-in-an-ai-first-world-4f20)

### 7-2. 現実の懸念・批判(建前との乖離)

- **人間が「受動的承認者(rubber stamp)」に堕すリスク**: AI-DLC 自身が「keep humans from becoming passive approvers」を設計目標に掲げること自体、この失敗モードが本質的リスクであることを裏づける。生成速度に検証が追いつかなければ、チェックポイントは形骸化する。
  - 出典: [GitHub awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)
- **学習/内省機構の欠如・理論的裏づけの薄さ**: 批判的レビュー(Peter Tilsen)は「context memory の実装詳細が不透明」「実装メカニクス(プロンプト・フォルダ構成・手順)偏重で理論的説明を欠く」と指摘。方法論としての深い理解と進化が妨げられるとする。
  - 出典: [Medium "AI-DLC: A critical, yet hopeful view"(Peter Tilsen)](https://medium.com/data-science-collective/the-ai-driven-development-lifecycle-ai-dlc-a-critical-yet-hopeful-view-edc966173f2f)
- **ドキュメント維持コスト**: 各フェーズで大量ドキュメントが生成されるが「作るより維持が難しく、コード変更に追随する仕組みがないとすぐ陳腐化する」。
  - 出典(二次): [Blogメモ「AIでの開発プロセスを考えてみた。AI-DLC、cc-sdd、GitHub Spec Kit」](https://blog.nightonly.com/2026/05/04/)
- **責任の所在(組織的アカウンタビリティ)**: AIエージェントは責任主体になれない。「誰がリスクを把握・管理し、何か起きたとき誰が責任を負うか」は自律エージェント社会の根本問題。AI-DLC は「humans remain responsible」と宣言するが、**具体的な責任分界(どの承認が誰の法的/契約的責任か)は方法論に規定がない**。
  - 出典: [JIPDECレポート「AIエージェントの実用化に向けた論点の整理」](https://www.jipdec.or.jp/library/itreport/20260422jipdecreport.html)
- **ハルシネーション/品質保証**: AIエージェントはハルシネーション・推論の欠如・バイアスを抱え、障害は「計画の順守失敗」と「新情報の捏造」に分かれデバッグが難しい。人間監視・データ改善・法整備が不可欠とされる。自己検証(AIがAIをテスト)には循環依存の懸念(テストの正しさを誰が保証するか)。
  - 出典: [アットマークIT「ハルシネーションか計画順守失敗か」](https://atmarkit.itmedia.co.jp/ait/articles/2604/27/news057.html)、[note「AIエージェントの限界」](https://note.com/umibenoheya/n/nf42544ecea20)
- **Operations フェーズの未成熟**: 現行は placeholder であり、運用・インシデント対応・セキュリティ運用は「将来の意図」に留まる。
  - 出典: [Micah Walter "Notes on exploring AI-DLC"](https://www.micahwalter.com/posts/notes-on-exploring-ai-dlc)

### 7-3. 日本企業固有の衝突(既存の決裁・品質保証・受発注構造との乖離)

<!-- 一次情報が薄い領域。以下は既存フェーズ1メモ(ウォーターフォール/ハイブリッド)との突き合わせによる解釈を含む。断定は避ける -->

- **多段稟議・決裁ゲートとの時間軸の不整合**: AI-DLC の bolt は「時間〜1日」で回るが、日本企業の稟議・DR(デザインレビュー)・投資決裁は週〜月単位で多段。高速な bolt サイクルに人間の決裁が構造的に追いつかず、「AIは回るが承認が詰まる」ボトルネックが予測される(解釈)。
- **品質保証部門(第三者検証)の位置づけ不在**: 日本の大手SIer/事業会社は独立QA部門・品質保証部による第三者レビューを制度化していることが多い。AI-DLC は QA を Mob(同席レビュー)に取り込む前提で、**独立第三者検証の制度と噛み合わない**(独立性 vs 同期協働のジレンマ)。
- **受発注構造と検収の二重性**: 日本のSI受託では、成果物レビューが「発注者検収」を兼ねる。AIが成果物を大量生成する AI-DLC では「何を検収対象とするか(コードか、意図か、プロンプトか)」「AI生成物の瑕疵担保責任は受注者か発注者か」が契約上未整備。
- **建前と実運用の乖離パターン(予測)**: 「AIが自律的に回る」を建前に導入しても、実運用では(1)承認を恐れる管理職が全チェックポイントで実質再レビュー→速度優位が消える、(2)逆に決裁が形骸化しAI生成物を無検証で通す→品質・セキュリティ事故、の両極に振れやすい。中庸(実効的なHuman-in-the-loop)を保つ組織能力が要る。
- 上記は一次情報が乏しく、本プロジェクト §9 の追加調査対象。既存の[ハイブリッド/アンチパターン調査メモ](./20260709-hybrid-antipatterns.md)、[ウォーターフォール調査メモ](./20260709-waterfall.md)と接続して深掘りする。

## 8. 考察(事実と分離)

- **AI-DLC の新規性の核は「粒度」と「同期」**: 従来SDLCの逐次ハンドオフを、(a) bolt による時間箱の極小化、(b) Mob による全ロール同期、(c) AIによる中間成果物の即時生成、で圧縮する。生産性主張(10-15x)の源泉はここにある。ただし数値は AWS 発の自己申告で、独立検証は未確認。
- **本プロジェクトにとっての To(理想)と From(現実)の橋渡し**: AI-DLC は「理想像」として使えるが、そのまま日本の組織へ載せると §7-3 の衝突が起きる。Process Compass の価値は、AI-DLC のチェックポイント(§5)を**日本の既存決裁ゲート・品質保証・検収に写像する変換表**を提供することにある(フェーズ3以降の設計論点)。
- **SDD(#27)との統合的整理が有効**: AI-DLC(組織プロセス層)⊃ SDD(仕様生成の技法層)⊃ 各エージェントツール(Kiro/Spec Kit/Q Developer)という入れ子で図解すると、読者の混乱を防げる。TDD メモと同様「粒度が違う概念を同一平面に並べない」方針を踏襲すべき。
- **最大の論点=「人が管理できる賢さ/速さの上限」**: AI-DLC の全設計は「人間が承認し責任を持つ」に依存する。だが承認の実効性は人間の検証帯域に律速される。ここが理想像の最大の脆弱点であり、本プロジェクトが現実解を出すべき中心。
