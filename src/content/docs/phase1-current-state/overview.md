## 目的

主に日本で行われている開発プロセスの現状を体系的に整理し、フェーズ3のギャップ分析の **From(現在地)** を定義する。

## 調査対象プロセス

- ウォーターフォール開発プロセス
- アジャイル開発プロセス
- スクラム開発プロセス
- テスト駆動開発(TDD)
- イベント駆動開発
- ドメイン駆動設計(DDD)
- 仕様駆動開発(SDD / Specification-Driven Development)
- AIDLC(AI-Driven Development Life Cycle)

## 調査観点

単なるプロセスの流れだけでなく、組織論まで踏み込んで整理する。

### ロール・組織

- ロールモデル(プロダクトオーナー、スクラムマスター、開発リーダー、セキュリティ代表者 など)
- 組織的な役割分担と責任の所在
- 第三者の専門部隊・社内の技術専門家チーム・CTO レビューの位置づけ

### ゲート・意思決定

- DR(デザインレビュー)・マイルストーン・決裁ゲートの考え方
- 「誰が承認したら事業として許可が出たことになるのか」という判断基準・品質基準
- 決裁では何を決裁して次に進めているのか
- V字モデル上でどこにゲートを設けているか

### 成果物・レビュー

- 各マイルストーンで作られるべきインプット/アウトプット(要件定義書、詳細設計書 など)
- 成果物を誰が・どういう形でレビューするか(レビュープロセス)

### 階層構造

プロセスは階層構造で整理する。全体プロセス → 各フェーズの中のワークフロー → 個々の作業。
例: DR1〜DR2 の間に行われる仕様設計の中のプロセス、セキュリティチェックの中の脆弱性診断 → トリアージ → 事業リスクと投資コストのバランス評価 → 施策選択 → バックログ化。

## 成果物

- 各プロセスの体系的なドキュメント(図解中心: シーケンス図、ロール図、ゲート図)
- プロセス間の比較表(ロール・ゲート・成果物・レビューの対応関係)

## 期間目安

3〜4ヶ月を想定しています。
