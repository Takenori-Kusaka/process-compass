## 背景

生成AIの能力は、人間が扱える範囲を超えて賢さを追求し始める段階に入りました。
しかし業務に組み込む上では「組織的にハンドリングできる賢さ」が実質的な上限になります。
モデルの賢さではなく、**私たちが扱いたい賢さとしての期待の上限**に近づいてきている今こそ、
生成AIを組み込んだ開発プロセスを体系的にまとめる好機だと考えています。

一方で、AIDLC(AI-Driven Development Life Cycle)は次のように語られがちです。

> AIが自律的に開発を回し、人は意思決定だけを行う。プロダクトオーナーとして動くチームだけが存在する。

しかし企業での開発には、組織的な責任の持ち方・品質保証のあり方が存在し、この理想像の単純な導入は困難です。
かといって何も定義せず、個人が現場で野良利用している状態を放置・助長してよいわけでもありません。

## 目的

1. **現行の開発プロセスを体系的に整理する** — ロールモデル、組織的役割、DR(デザインレビュー)・決裁ゲート、マイルストーン成果物、レビュープロセス、品質基準まで細かく
2. **AIDLC・生成AI前提の理想プロセスを整理する**
3. **両者を突合し、現実的な統合プロセスを策定する** — 「どのフェーズで、誰が、どう生成AIを使うと良いか」を具体化
4. **プロセスの実装・運用方法まで落とし込む** — Git戦略、CI/CDゲート、AI実行環境、運用作業

最終的には、各企業の品質管理部門・プロセス部門に**提案書として提出し、承認・運用してもらえるレベル**の具体性を目指します。

## 最終ゴール: プロセス提案ツール

このリポジトリの GitHub Pages を、単なるドキュメント集ではなく**開発プロセス支援ツール**として確立します。

- **入力**: アサイン可能なメンバー、稼働時間、チーム体制、プロダクトの期待品質、事業フェーズ など
- **出力**: そのチームに最適な開発プロセスの提案
- **運用**: 提案をベースにチームが修正・採択し、具体的なプロセス運用の実装方法・運用方法を策定できる
