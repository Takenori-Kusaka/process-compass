プロセスは導入して終わりではなく、回し続けるための**定常作業**が発生します。このページはその作業カタログです。「誰が・どの頻度で・何をするか」を明示し、運用が特定個人の善意に依存しない状態を作ります。

## 定常作業カタログ

### 日次〜随時

| 作業 | 担当 | 内容 |
| --- | --- | --- |
| ゲート判定 | 各判定者 | 期限内の判定(要件合意48h・仕様承認24h・独立レビュー2営業日) |
| インシデント一次対応 | 開発者+AI | 検知・一次分析はAI、対処判断は人。コア機能は理解保持者が対応 |
| 負債の記録 | 開発者 | 生成過程で見つけた妥協をその場で台帳へ(ためない) |

### 週次

| 作業 | 担当 | 内容 |
| --- | --- | --- |
| ゲート健全性の確認 | 価値責任者 | 滞留時間・差し戻し率・エスカレーション数の確認([メトリクス](/process-compass/phase6-operation/metrics/)) |
| 文脈の書き戻し確認 | 文脈オーナー | 今週の学び(レビュー指摘・障害・質問)で文脈化すべきものの拾い上げ |
| バックログの棚卸し | 価値責任者 | 優先順位の見直し(負債返却枠を含む) |

### 月次

| 作業 | 担当 | 内容 |
| --- | --- | --- |
| 負債返却の計画 | 価値責任者+開発者 | 台帳のトリアージと返却枠への割り当て([負債返却プロセス](/process-compass/phase6-operation/debt-payback/)) |
| CI ゲートの見直し | 技術判断者 | ルール化候補(同種指摘3回)の吸い上げ、誤検知ルールの調整 |
| 文脈の陳腐化チェック | 文脈オーナー | 停滞ファイルの棚卸し(自動一覧から) |
| アクセス・権限の確認 | 技術判断者 | AIエージェントの権限設定・シークレットの棚卸し |

### 四半期

| 作業 | 担当 | 内容 |
| --- | --- | --- |
| プロセスの振り返り | 全ロール代表 | メトリクスに基づくプロセス自体の適応([改善サイクル](/process-compass/phase6-operation/improvement-cycle/)) |
| モデル更新の検証・切り替え判断 | 技術判断者 | 新モデルの検証と切り替え(随時発生にも対応) |
| コア指定の見直し | 価値責任者+技術判断者 | 事業の変化に応じたコア/非コアの再判定 |
| 担当表の総点検 | 文脈オーナー | ロール担当表・CODEOWNERS の実態との突合(異動対応) |

## 運用の原則

- **作業はカレンダーに載せる**: 「気づいた人がやる」にしない。定常作業は繰り返し予定・自動リマインドとして仕組み化する
- **AIに任せられる部分は任せる**: 集計(メトリクス・エビデンス)・一覧化(陳腐化候補・ルール化候補)・起案(書き戻し案)をAIが実行し、判断だけを人が担う——運用でもRACIの原則は同じ
- **空席を作らない**: 担当ロールが異動・退職で空いたら、担当表の更新を最優先にする(空席の放置が運用崩壊の起点)

## 運用開始時のチェックリスト

導入プロジェクトがリリースを迎えて運用に入るとき、次を確認します。

1. 運用引き継ぎ文書(テンプレ5)が出荷判定を通過している
2. 上記カタログの各作業に担当者の名前が入っている(ロール名だけの状態にしない)
3. メトリクスの自動集計が動いている(手集計の運用は続かない)
4. コア機能の理解保持者が2名以上いる(1名は単一障害点)
